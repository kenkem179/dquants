# PIPELINE CONTRACT — hypothesis → develop → validate → deploy, made MT5-consistent

_Authored 2026-06-16. Supersedes the implicit "trust the engine" workflow that promoted 3 EAs which all
lost in MT5. Companion to `optimization/HONEST-AUDIT-2026-06-16.md`. Primary target: **Monster**._

## 0. The one defect this fixes
The old pipeline validated candidates against the **C++ engine**, but never proved the engine equals
**MT5**. "Validated" meant "the engine likes it," not "MT5 likes it." Result: configs that the engine
scored PF>1.2 lost money in MT5. This contract inserts the missing step — **an MT5 parity gate that a
candidate must pass before it can be called validated** — and bans the practices that caused the drift.

## 1. The four stages and their hard boundaries

```
HYPOTHESIS → DEVELOP → VALIDATE → DEPLOY
   (research)  (engine)  (MT5 parity + OOS gate)  (ship .ex5)
```

A candidate may only advance when it satisfies the **gate** at the boundary. No gate skip, ever.

## 2. ENGINE RULE (closes the bar-vs-tick sign flip)
- **Only the tick engine produces P&L or ranking numbers.** Binaries:
  `cpp_core/build/monster_backtester`, `.../mastervp/...`, `cpp_core/build/kenkem/tick_backtester`.
- **The bar-OHLC engine is QUARANTINED.** It disagrees with MT5 on the *sign* of P&L. It may be used
  only for fast feature/EDA exploration — **never** for a backtest, sweep, or any promotable number.
  `optimize_*.py` must point at a tick-engine binary or be considered invalid.
- Every P&L claim in any doc must name the engine + binary that produced it. Unattributed PF = invalid.

## 3. CONFIG RULE (closes param contamination)
- The engine **must refuse** to honor any `.set` key the EA hardcodes (not an MQL5 `input`). Already
  enforced: `is_ea_locked_key()` / `monster_non_input_keys()`. Audit: `kenkem_parity/PARAM_SURFACE_AUDIT.md`.
- **Class-A (honorable)** params = real MQL5 `input`s → sweepable. **Class-B (locked)** = hardcoded →
  engine warns once + keeps the EA value → never sweepable. A sweep that moves a Class-B key is void.
- Every `best_*.set` must be regenerated by a tick-engine sweep over Class-A only, or it is untrusted.

## 4. VALIDATE GATE — the MT5 parity check (manual MT5 for now)
A candidate config is **VALIDATED** only when an engine run and an MT5 run on the **same** .ex5 + .set +
ticks + window agree trade-by-trade. Until headless automation exists (future), MT5 is run **manually**.

**Loop (per candidate):**
1. Engine: run tick engine with the config → exports `trades_<sym>.csv` + per-bar `parity_<sym>.csv`
   (entry bar/time, dir, entry px, SL, TP, exit time, exit reason, P&L, lots).
2. MT5 (manual): load the **same** .ex5 with the **same** .set, same symbol/TF/dates, "Every tick based
   on real ticks", same deposit/leverage/spread source. Run. The EA's `Parity/` hooks export the
   matching CSVs to `../kenkem/Tester/Agent-127.0.0.1-3000/MQL5/Files/<strategy>/`.
3. Diff: run `research/validation/parity_diff.py` (TO BUILD) → aligns the two trade lists and reports:
   - **trade-count delta**, unmatched entries each side
   - per-matched-trade: dir match, entry-bar Δ, SL/TP Δ, exit-reason match, P&L Δ
   - **net P&L Δ %**, PF each side.

**PASS criteria (tolerance):** entries match 1:1 (≤1-bar lag on ≤5% of trades), exit reasons match,
**net P&L within ~0.5–1%**. Any structural mismatch (different trade count, opposite direction, wrong
exit path) = **FAIL → it's an engine-fidelity bug, fix the engine, do not promote.**

## 5. DEPLOY GATE
Ship a config only if **both** hold:
1. **Parity PASS** (§4) on the recent OOS window.
2. **MT5 net beats the incumbent** — i.e. the user's ORIGINAL `KenKemExpert` (XAU M1, PF 1.62, +24%) —
   on the same recent OOS window. A port that doesn't beat the thing already running doesn't ship.

"Compiles clean" is **not** a gate. It says nothing about profit. Never present it as readiness.

## 6. MONSTER — current truth & the fidelity work (target strategy)
- **Stale diagnosis retired:** "Monster makes 0 trades / MQL5 keys off broker VOLUME=0." The deployed
  `mql5/experts/KK-Monster/Engine.mqh::MonNearNet` is a body-direction-proxy port of the C++
  `tf_net_near_at` (NOT broker volume). On 2026-06-16 it fired **2,576** entries (XAU M1) — plenty.
- **Live divergence:** engine claims full PF ~1.23–1.32; MT5 today is SL-dominated (~18% TP) and
  unprofitable. So engine and MT5 disagree on **economics**, not signal firing. Suspect causes, in order:
  1. **Cost model** — engine spread/commission/slippage likely lighter than the Exness feed in MT5;
     at 1000s of trades this alone flips PF. *Verify engine costs == MT5 symbol spec first.*
  2. **Exit geometry** — engine vs EA may close on different rules (trail vs managed) → win%/PF gap.
  3. **Contaminated `.set`** — confirm the tested .ex5/.set use only Class-A params (§3).
- **Net-volume parity sub-check:** there are TWO net implementations — `tf_net.hpp::tf_net_near_at`
  (the gate net, ported to `MonNearNet`) and `tick_engine.hpp::build_net_flow_` (exit net). Confirm the
  EA uses the same one the engine's gates use; an off-by-one in the `[1]`-shift read poisons every gate.

**Monster next action (after this spec):** run the §4 loop once on Monster XAU M1 recent OOS — engine
export + (manual) MT5 run + the new `parity_diff.py` — and read the first real trade-level diff. That
diff tells us which of 6.1/6.2/6.3 is the culprit. Fix the engine to match MT5, re-run until PASS.

## 7. What's automated now vs later
- **Now:** stages 1–3 automated (engine + diff tool); **MT5 run is manual** (user). The diff tool +
  export contract make the manual run cheap and unambiguous.
- **Later (deferred by user):** headless MT5 tester via a scriptable `.ini` (`terminal64.exe /config:`
  + `ShutdownTerminal=1` + report export) wired into stage 2 → fully automatic gate. Wine path proven
  by `scripts/compile_mql5.sh`.

## 8. Definition of done for "the pipeline is consistent"
Monster (then MasterVP, then KenKem) passes §4 parity on recent OOS, and the only configs that ever
reach `mql5/experts/` are those that cleared §5. At that point an engine PF is a *reliable predictor* of
the MT5 PF, and hypothesis→deploy is trustworthy end-to-end.
