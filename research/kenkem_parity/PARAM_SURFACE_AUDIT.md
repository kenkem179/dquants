# KenKem param-surface audit — the trust fix

**Date:** 2026-06-16. **Why this exists:** the user's core complaint — *"the original EA is profitable, but
the C++-optimized configs lose in MT5."* This audit found the systemic cause and fixed it structurally.

## The root cause (quantified)

dquants `kenkem_config.hpp::apply_kv` accepted **223** `.set` keys. Cross-referencing every key against the
real KenKem EA source (`kenkem/MQL5/Experts/KenKem/Config/InputParams.mqh`, 410 `input`s + 111 hardcoded
globals) split them into three classes:

| Class | Count | Meaning | Sweepable? |
|---|---:|---|---|
| **A — EA-LOCKED** | 21 | EA HARDCODES these (NOT an `input`); MT5 **silently ignores** any `.set` value | ❌ never |
| **B — real EA input** | 173 | genuine EA `input`; MT5 honors it | ✅ yes |
| **C — synthetic / dquants-only** | 29 | no EA equivalent (8 are session aliases → locked; rest are dquants extensions) | parity: must stay neutral |

**The CLASS A keys are the bug.** Because MT5 hardcodes them, any dquants sweep that *moved* one produced a
config the EA cannot reproduce. When that "optimized" config is loaded into MT5, MT5 runs its hardcoded
value instead → different indicators → different trades → the profit evaporates. This is precisely the
ADX_LEN bug that inverted the KenKem verdict last session (PF 0.90 "losing" was really ADX(9); the EA always
ran ADX(14)).

### Contamination found across the repo (scan of all 73 `.set` files)
- **51** KenKem sweep-result sets override `ADX_LEN` (to 8/9/10/11/13/15) — EA hardcodes **14**.
- **51** override `RSI_LEN` (to 8/9/10/11/16/17/19/20) — EA hardcodes **14**.
- Even the **parity reference sets** still carried `RSI_LEN=20` (xau) / `RSI_LEN=11` (btc) — a live parity bug
  (ADX_LEN was fixed last session; RSI_LEN was missed).

⟹ **Every KenKem "best_*" sweep result in `research/optimization/` is invalid** — it was tuned on indicator
periods the EA never runs. They are quarantined (see below) and must be regenerated by a clean sweep.

## The fix (structural, in `kenkem_config.hpp`)

`apply_kv` now calls `is_ea_locked_key()` and **refuses** any CLASS A key: it keeps the EA's hardcoded value
(which the C++ struct default already equals) and prints a one-time warning. The engine is now *physically
incapable* of honoring a parameter MT5 can't — no future sweep can silently diverge. Locked test:
`test_ea_locked_keys_ignored`.

### CLASS A — the 21 locked keys (EA value in parens)
- `ADX_LEN`(14), `RSI_LEN`(14)
- `ICHIMOKU_TENKAN`(9), `ICHIMOKU_KIJUN`(26), `ICHIMOKU_SENKOU`(52)
- `USE_CONVICTION_SCORING_E1`(true), `_E2`(true), `_E4`(true)
- `USE_HTF_VETO_E1`(false), `_E2`(false), `_E4`(false)
- `USE_ICHIMOKU_E1`(true), `_E2`(false), `_E4`(false)
- `IGNORE_VALID_SESSIONS`(false)
- `JAPAN_START`(0), `JAPAN_END`(330), `LONDON_START`(500), `LONDON_END`(930), `NY_START`(1200), `NY_END`(1500)
  - plus the 6 `*_SESSION_START/END` aliases for the same hardcoded windows.

### CLASS C — dquants-only extensions (must stay NEUTRAL for MT5 parity; all default off)
`MAX_ENTRIES_PER_DAY`(0), `SERVER_GMT_OFFSET`(0), `E5_REQUIRE_TREND_CORE`(models EA), and the 20 `PM_*`
shared-ProfitManager toggles (the KenKem EA has no ProfitManager module — these belong to the future unified
common executor, not to KenKem-vs-MT5 parity). Defaults are parity-safe; a sweep that enables them is
"research beyond the EA", not a reproducible MT5 config.

## The bug is SYSTEMIC — it hits all three strategies (added 2026-06-16)

The same audit was run for KK-MasterVP and KK-MasterVP-Monster. **Both expose EA-hardcoded params, and
both have contaminated `best_*` sets:**

| Strategy | C++ keys | EA-hardcoded keys exposed | best-set contamination found | Fix |
|---|---:|---|---|---|
| **KenKem** | 223 | 21 | `ADX_LEN`/`RSI_LEN` in 51 sets + parity sets | `is_ea_locked_key()` refuses them |
| **MasterVP** | 110 (shared) | 12 | `best_mastervp_{btc,xau}.set`: `InpAtrLen=11/15` (EA 14), `InpVpBins=21/49` (EA 30), `InpVaPct≈75` (EA 70) | `non_input_keys()` already skipped most via `mimic_mt5_noninput=true`; **`InpAtrLen` was missing → added** |
| **Monster** | 154 | 15 | `best_monster_{btc,xau}.set`: `InpNodeDecay/NeutralBand/Saturation/TouchAtr` overrides (all hardcoded) | new `monster_non_input_keys()` + `load_set` now refuses them |

Note the lock is **per-strategy**: e.g. Monster hardcodes the node params (`InpNodeDecay`…) that MasterVP
genuinely tunes as inputs, so each engine carries its own locked set. MasterVP's `InpVpBins=49` was already
inert (the backtester ran `mimic_mt5_noninput=true`), but `InpAtrLen` leaked until this fix.

**Consequence:** the entire `research/optimization/best_*` set for KenKem (51) **and** MasterVP (2) **and**
Monster (2) is unreliable for an MT5 production decision — each was tuned with ≥1 param MT5 can't honor.
They must be regenerated by a clean sweep (tick engine + honorable params only) before ranking strategies.
